# Marketplace And Vendor Gap Review

This Phase 20 review records what remains missing before Open Transit RT could be described as marketplace/vendor equivalent.

The current repository is open-source software plus documentation and evidence workflows. It is not a hosted SaaS product, agency-endorsed service, procurement-ready vendor package, or approved marketplace offering.

## Gap Summary

| Area | Current evidence | Remaining gap |
| --- | --- | --- |
| Support model | Runbooks and docs exist for local demo and small-agency pilot operations. | Named support owner, response channels, escalation process, coverage hours, and support commitments are not established. |
| SLA/SLO commitments | Suggested SLO concepts and metrics foundations exist. | Contractual uptime, feed freshness, incident response, validation cadence, and reporting commitments are not defined or staffed. |
| Procurement docs | README, tutorials, runbooks, evidence docs, and security policy exist. | Procurement package, service description, pricing, insurance/security questionnaires, accessibility statements, and contract terms are not present. |
| Onboarding package | Local agency app flow, demo flow, device onboarding helper, and pilot operations runbook exist. | Production onboarding checklist for real agency identity, devices, GTFS ownership, DNS, license approval, staff training, and consumer submissions is incomplete. |
| Maintenance plan | Versioned migrations, Make targets, validators, and pilot helper scripts exist. | Release cadence, upgrade policy, dependency patch process, validator refresh cadence, and long-term maintenance ownership are not formalized. |
| Hardware/device support strategy | Authenticated telemetry ingest and device-token binding exist. | Approved hardware list, BYOD device requirements, installation guide, replacement process, and field support model are not defined. |
| Incident response commitments | Security policy and evidence redaction policy exist; request IDs and logs exist in runtime. | Production incident process, severity definitions, notification windows, customer communication templates, and post-incident review commitments are not established. |
| Customer support and training | Docs and tutorials support self-guided evaluation. | Operator training materials, support playbooks, helpdesk process, and role-specific agency training are not packaged. |
| Production hosting/managed service strategy | OCI pilot runbook and evidence show one resource-constrained pilot path. | Managed service architecture, tenancy model, backups, monitoring, scaling, data retention, disaster recovery, and operations staffing are not defined as an offering. |
| Consumer/aggregator onboarding | Prepared Phase 20 packets exist for seven targets. | Official submission paths are not verified, no submissions are evidenced, and no target acceptance is evidenced. |

## Required Before Vendor-Like Positioning

Before claiming vendor-like adoption readiness, the project would need:

- an explicit service/support owner and contact path;
- published support scope and response targets;
- deployment and managed-hosting options with security and operations commitments;
- agency onboarding package with staff roles and required approvals;
- device procurement/BYOD guidance;
- maintenance and upgrade policy;
- incident response process;
- consumer-submission operating procedure with redacted evidence retention;
- legal/procurement artifacts appropriate for the target agencies or programs.

## Claim Boundary

This review is a gap list only. It does not claim marketplace listing, vendor approval, agency endorsement, hosted SaaS availability, or production support commitments.
